# 双数据源改造（AKShare 主 + tushare 备）— 设计文档

## 1. 目标

将数据源从单一 tushare 改造为 AKShare 主、tushare 备的双源架构。AKShare 偶发限频或接口失效时自动回退到 tushare，保证系统可用性。同时移除已确认的死代码（moneyflow 链路），并增加盘后自动更新机制。

## 2. 动机

原 tushare 单源方案存在两个问题：

1. **成分股列表拿不到**：2000 积分下 `pro.index_weight` / `pro.index_member` 不可用，被迫硬编码 49 只成分股列表。成分股调整后需手动维护。
2. **单点故障**：tushare API 异常时整个系统无法加载数据，无回退路径。

AKShare 优势：免费、无需 token、`ak.index_stock_consindex("399975")` 能直接拿到动态成分股列表。劣势：爬虫式接口，偶发限频和网页改版。双源架构取两者之长。

## 3. 架构

```
fetch_index_daily / fetch_stock_daily / fetch_index_constituents
        │
        ├─ 主源 AKShare (akshare)
        │     ├─ 成功 → 归一化 → 返回
        │     └─ 失败 ↓
        ├─ 备源 tushare
        │     ├─ 成功 → 归一化 → 返回
        │     └─ 失败 ↓
        └─ 兜底（仅成分股：硬编码 INDEX_CONSTITUENTS）
```

### 3.1 单位归一化层

两个数据源的字段名和单位不一致，必须在 `data_fetcher.py` 内部统一到 **tushare 口径**（与历史缓存一致），对外完全透明：

| 数据源 | vol 单位 | amount 单位 | 归一化换算 |
|---|---|---|---|
| AKShare 个股 `stock_zh_a_daily` | 股 | 元 | ÷100 / ÷1000 |
| AKShare 指数 `stock_zh_index_daily` | 股 | — (无 amount) | ÷100 / NaN |
| tushare `pro.daily` / `pro.index_daily` | 手 | 千元 | 原样 |

归一化后 schema：`ts_code, trade_date, open, high, low, close, vol(手), amount(千元)`。

**为什么用 tushare 口径作目标**：历史缓存（2008 年起）是 tushare 拉取的，单位是手/千元。增量拉取的新数据若单位不一致，会让 MA250、OBV 等长周期指标在拼接点跳变。统一到 tushare 口径可零改动复用历史缓存。

### 3.2 字段映射

| 统一字段 | AKShare 个股源 | AKShare 指数源 | tushare |
|---|---|---|---|
| ts_code | 由 symbol 拼接 (`600030.SH`) | 常量 `399975.SZ` | 原样 |
| trade_date | `date` | `date` | `trade_date` |
| open/high/low/close | 同名 | 同名 | 同名 |
| vol | `volume / 100` | `volume / 100` | `vol` |
| amount | `amount / 1000` | NaN | `amount` |

### 3.3 接口选择

- **个股**：用 `ak.stock_zh_a_daily`（新浪源，`adjust=""` 不复权）而非 `ak.stock_zh_a_hist`（东财源）。新浪源字段名更接近 tushare 且更稳定；东财源偶发 `RemoteDisconnected`。
- **指数**：`ak.stock_zh_index_daily(symbol="sz399975")`。
- **成分股**：`ak.index_stock_cons_csindex(symbol="399975")`，返回 49 只当前成分股。

## 4. 缓存与盘后更新

### 4.1 缓存策略

```
force=False 命中缓存的条件（全部满足）：
  1. 文件存在
  2. mtime 在 12 小时内（_is_fresh）
  3. 不需要盘后更新（_needs_eod_update 返回 False）

否则 → 走增量或全量拉取
```

### 4.2 盘后自动更新 `_needs_eod_update`

工作日 15:00 收盘后，若缓存最新交易日 < 今天，自动触发增量拉取当天收盘数据。

```python
def _needs_eod_update(path):
    now = datetime.now()
    if now.hour < 15:           # 盘中不触发
        return False
    if now.weekday() >= 5:      # 周末不触发
        return False
    last = pd.read_csv(path)['trade_date'].max().date()
    return last < now.date()
```

### 4.3 增量拉取

缓存存在时，从缓存最新日期 +1 天起拉新数据，拼接去重后落盘。增量分支内加日期守卫：缓存最新日期已 ≥ 今天 → 直接返回，避免重复拉取（force=True 也遵守）。

### 4.4 行为对照

| 场景 | 行为 |
|---|---|
| 盘中（<15:00）打开 | 看 mtime 新鲜度，不重复拉 |
| 工作日 15:00 后首次打开 | 自动拉今日收盘数据 |
| 周末打开 | 不触发（无新数据）|
| 缓存已是今天数据 | 直接返回缓存（不拉 API）|
| 非交易日（节假日）盘后 | 触发拉取，源返回空 → 异常 → 用缓存（无副作用）|

## 5. moneyflow 链路废弃

改造前 `fetch_moneyflow` 调用 tushare `pro.moneyflow_dc` 拉资金流数据。审查确认是死代码：

- `app.py` 仅将结果存入 `session_state.stocks_moneyflow`，全项目无任何消费点
- `backtest.py:152` 因 `index_moneyflow` 恒为 None 永远 `continue` 跳过
- 个股回测 `backtest.py:493` 显式跳过 `requires_moneyflow` 规则
- 指标 `main_force_inflow` / `margin_stabilize` 标记 `requires_moneyflow: True` 但从未实际触发

改造内容：
- 删除 `data_fetcher.fetch_moneyflow`
- 删除 `app.py` 的 `fetch_moneyflow` import、`stocks_moneyflow` session key、拉取循环
- 保留 `app.py:238` 的 `'index_moneyflow': None` 占位（让 backtest.py 跳过分支自然生效）
- 保留 `indicators.py` 的 `requires_moneyflow` 注册项（标注但永不触发，便于未来恢复）

## 6. 文件改动

| 文件 | 改动 |
|---|---|
| `data_fetcher.py` | 重写为双源架构 + 归一化层 + 盘后更新，删除 `fetch_moneyflow` |
| `app.py` | 删除 moneyflow 相关 import 和调用（5 行）|

`indicators.py` / `backtest.py` / `screener.py` / `detect_cycles.py` 零改动。对外 schema 不变，历史缓存继续可用。

## 7. 验证

- AKShare / tushare 同一交易日 OHLC + vol 完全一致（单位归一化后）
- AKShare 成分股 49 只对上原硬编码列表
- 双源回退：AKShare 抛错 → tushare 接管；全失败抛聚合异常
- 成分股三层回退：AKShare → tushare → 硬编码
- 真增量拉取（截断缓存）：拼接点 vol 量级连续，单位一致
- `screener.py --calc-only` 全链路 49 只重算成功
- 工作日 15:00 后 `force=False` 自动拉取今日数据

## 8. 已知限制

- AKShare 新浪指数源无 `amount` 字段（填 NaN）。已确认 `amount` 全项目仅出现在文档字符串，无任何计算消费，无影响。
- AKShare 是爬虫式接口，未来网页改版可能失效。回退到 tushare 保证可用性，但 tushare 2000 积分拿不到成分股列表（届时回退硬编码）。
